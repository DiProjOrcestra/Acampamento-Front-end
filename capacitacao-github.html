🐙 Git & GitHub: Controle de Versão

Fundamentos de Versionamento e Colaboração em Equipe

Antes de escrever o código do seu portfólio, é fundamental entender como versionar, proteger e compartilhar o seu projeto com a equipe.

📖O que é Git e GitHub?

O que é Git? (O Motor Local)

O Git é um sistema de controle de versão instalado no seu computador. Ele é a ferramenta que rastreia cada alteração que você faz no código.

  • Local: Funciona na sua máquina, mesmo sem internet.
  • Máquina do tempo: Permite que você salve estados do seu projeto (snapshots) e volte para eles caso algo dê errado.
  • Rastreabilidade: Ele sabe exatamente quem alterou cada linha de código e quando isso aconteceu.

O que é GitHub? (A Nuvem Colaborativa)

O GitHub é uma plataforma online que hospeda repositórios Git. Ele não é o Git, mas sim um serviço na nuvem que permite a interação entre vários desenvolvedores.

  • Remoto: É onde o código "vive" para que o time inteiro possa acessá-lo.
  • Colaboração: É aqui que as ferramentas como Pull Requests e Issues acontecem.
  • Segurança: Funciona como um backup do seu trabalho, garantindo que ninguém perca o progresso se o computador de alguém quebrar.

A analogia definitiva

Pense no Git como o seu editor de texto ou o seu ambiente de desenvolvimento (a ferramenta que gera o conteúdo) e no GitHub como um Google Drive ou Dropbox especializado para programadores (o lugar onde você armazena e compartilha esse conteúdo com o mundo).

⚙️Configuração Inicial

Antes de usar o Git, você precisa configurar sua identidade. Essa informação ficará registrada em todos os commits que você fizer.

Configurando seu Usuário Globalmente

Abra o terminal e execute os seguintes comandos, substituindo pelos seus dados reais:

git config --global user.name "Seu Nome"
git config --global user.email "seuemail@exemplo.com"

Esse passo é indispensável para que o Git vincule diretamente cada linha de código modificada à sua conta, garantindo que o seu histórico de contribuições apareça corretamente no seu perfil do GitHub.

🎯 Como verificar se a configuração deu certo?

Depois de rodar os comandos acima, execute os seguintes comandos no terminal para conferir se os dados foram salvos corretamente na sua máquina:

git config user.name
git config user.email

Se o terminal retornar exatamente o seu nome e o seu e-mail embaixo de cada comando, a vinculação foi feita com sucesso e você já está pronto para realizar seus commits.

📥Clonagem de Repositório

O que é um Clone?

O comando git clone é o ponto de partida para colaborar em qualquer projeto existente. Ao contrário de um simples "download" de arquivos, o clone baixa o projeto completo, incluindo todo o seu histórico de versões e branches. 

git clone [URL-do-repositorio]

Ao executar esse comando, o Git:

  • Cria uma cópia exata do repositório remoto na sua máquina.
  • Configura automaticamente a conexão (chamada de origin) entre o seu repositório local e o GitHub.
  • Permite que você comece a trabalhar imediatamente, mantendo a capacidade de sincronizar suas futuras alterações com a equipe.

📤Sincronização de Código

Fluxo de Commits e Sincronização

Nesta etapa, vamos registrar o seu progresso no projeto. O objetivo é transformar as modificações locais em um histórico organizado e seguro no GitHub.

O fluxo de trabalho depende de como o projeto começou:

# Cenário A: Se começou do zero na sua máquina
git init                  # 0. Inicializa o repositório (apenas 1x)
git add [arquivos]        # 1. Prepara as alterações
git commit -m "mensagem"  # 2. Registra o snapshot local
git push origin [branch]  # 3. Sincroniza com o GitHub

# Cenário B: Se você usou git clone
# O git init não é necessário, o repositório já está pronto!
git pull origin [branch]  # 0. Atualiza o código com as mudanças da equipe
git add [arquivos]        # 1. Prepara as alterações
git commit -m "mensagem"  # 2. Registra o snapshot local
git push origin [branch]  # 3. Sincroniza com o GitHub

Notas importantes:

  • git init: Necessário apenas para transformar uma pasta nova em um projeto versionável. Se você clonou um projeto, a pasta .git já existe.
  • git pull: Fundamental em projetos clonados. Ele traz as atualizações dos seus colegas para sua máquina, evitando conflitos antes de você enviar seu próprio trabalho.
  • git add: Move as alterações para a Staging Area (o filtro antes do commit).
  • git commit: Cria o "snapshot" (foto) do código.
  • git push: Transfere seus commits locais para o servidor.

Entendendo o git add e o destino do push

Antes de registrar um commit, utilizamos a Staging Area para filtrar o que será enviado:

  • git add . : Adiciona todas as modificações da pasta.
  • git add nome-do-arquivo.ext : Adiciona apenas um arquivo específico.

Sobre o git push: nem sempre o destino será a main. Em um fluxo de trabalho profissional, você fará o push para a branch na qual está desenvolvendo sua funcionalidade (ex: feature/login).

O comando git push origin [nome-da-sua-branch] garante que suas alterações subam para a branch correta no GitHub, mantendo o ambiente de produção (main) protegido até que sua feature seja revisada e aprovada.

Dica: Utilize o git status para verificar o estado dos seus arquivos e git branch para confirmar em qual ramificação você está antes de enviar.

⚠️ Boas Práticas para evitar conflitos

  • Sempre rode git pull antes de começar: Se você não estiver com a versão mais recente da branch, o GitHub pode rejeitar o seu push, gerando conflitos que levam tempo para resolver.
  • Commits curtos e atômicos: Não tente subir 5 funcionalidades de uma vez. Faça um commit por funcionalidade ou correção. Isso torna a revisão do código muito mais fácil para os seus colegas.
  • Cuidado com o que sobe: Nunca suba arquivos de configuração locais (como senhas, chaves de API ou pastas de bibliotecas pesadas como node_modules).

🛠️Comandos de Manutenção

Monitoramento e Diagnóstico

Comandos essenciais para saber o que está acontecendo no seu ambiente de trabalho.

  • git status : Exibe o estado atual (quais arquivos foram alterados, quais estão na Staging Area). Use sempre antes de qualquer commit.
  • git log --oneline : Mostra o histórico recente de commits de forma compacta e fácil de ler.
  • git branch : Lista as branches do repositório e mostra em qual você está trabalhando atualmente.

Gestão de Alterações Temporárias

Ferramentas para organizar seu fluxo sem perder trabalho inacabado.

  • git stash : "Guarda" suas alterações atuais em um local temporário. Útil quando você precisa trocar de branch urgente para um hotfix, mas não quer commitar código incompleto.
  • git stash pop : Traz de volta as alterações que estavam guardadas pelo stash.

⚠️ Quando as coisas ficam estranhas...

  • git diff : Mostra exatamente o que mudou linha a linha em cada arquivo antes de você fazer o add.
  • git checkout . : CUIDADO! Descarta todas as alterações feitas em arquivos que ainda não foram commitados. Use apenas se você quiser apagar tudo e recomeçar da última versão salva.

🔀Gerenciamento de Ramificações

O que são Branches?

Branches (ramificações) são linhas de desenvolvimento isoladas. Elas permitem que você trabalhe em novas funcionalidades, correções de bugs ou experimentos sem afetar o código estável do projeto (geralmente na main). 

git branch                    # Lista as branches locais
git branch nome-da-branch     # Cria uma nova branch
git checkout nome-da-branch   # Alterna para a branch desejada
git checkout -b nome-da-branch # Atalho: Cria e já alterna para a branch

Fluxo de Trabalho Recomendado

Para manter um projeto organizado em equipe, adote estas boas práticas:

  • Proteja a main: Nunca faça commits diretos na branch principal. Ela deve conter apenas código testado e estável.
  • Nomenclatura Padrão: Utilize prefixos para identificar o tipo de tarefa:
    • feature/nome-da-funcionalidade
    • fix/correcao-de-bug
    • docs/documentacao
  • Ciclo de Vida: Assim que sua tarefa na branch estiver concluída e testada, o próximo passo é solicitar a integração (via Pull Request) para a branch principal.

⚠️ Atenção ao Alternar (Checkout)

Antes de trocar de branch, garanta que seu ambiente está limpo ou suas mudanças estão guardadas:

  • O Git impedirá a troca se houver conflitos com arquivos que você modificou localmente.
  • Se precisar trocar de branch rapidamente sem finalizar a tarefa atual, use o git stash para guardar o progresso e o git stash pop para retomar depois.

📝Conventional Commits

Por que padronizar commits?

O Conventional Commits é um padrão que cria um histórico legível tanto para humanos quanto para máquinas. Com ele, você consegue gerar changelogs automáticos e identificar rapidamente o impacto de cada mudança no sistema. 

<tipo>(<escopo>): <descrição>

Estrutura recomendada

Utilize os tipos abaixo para categorizar suas alterações:

  • feat: Nova funcionalidade (feature).
  • fix: Correção de um erro (bug).
  • docs: Mudanças apenas na documentação.
  • refactor: Alteração de código que não corrige bug nem adiciona funcionalidade.
  • test: Adição ou correção de testes.
  • style: Ajustes de formatação, espaços, etc. (sem alterar lógica).

Exemplos reais:

feat(auth): adiciona validação de token JWT
fix(header): corrige alinhamento do menu mobile

⚠️ Erros comuns

  • Mensagens vagas: "corrige coisas", "ajustes", "update" — isso não diz nada sobre o que foi alterado.
  • Escopo gigante: Se o seu commit precisa de muitos tipos diferentes para ser explicado, talvez ele esteja grande demais. Considere quebrá-lo em commits menores (atômicos).
  • Falta de imperativo: O padrão é usar o modo imperativo ("Adiciona" em vez de "Adicionado").

🔀Pull Requests (PRs)

O que é um Pull Request?

O Pull Request é o convite formal para que sua equipe revise e integre o código que você desenvolveu na sua branch para a branch principal (como a main).

É o espaço onde:

  • Você explica o que foi feito e por que.
  • Seus colegas podem comentar, sugerir melhorias ou encontrar bugs antes do código entrar no projeto.
  • A equipe garante que o novo código segue os padrões do projeto.

Como criar um PR profissional

Um bom Pull Request acelera a revisão e evita retrabalho. Siga este padrão:

  • Título claro e direto: Ex: feature: implementa sistema de autenticação.
  • Descrição detalhada: O que mudou? Como testar? Se houver prints, adicione-os.
  • Linked Issues: Se o seu projeto usa *Issues*, mencione qual problema você está resolvendo (ex: Closes #12).

⚠️ Regras de Ouro na Revisão

  • Seja construtivo: Críticas devem ser sobre o código, nunca sobre a pessoa.
  • O autor resolve: Se pedirem alterações (requested changes), é seu papel fazer as correções e dar um "push" novamente para atualizar o PR.
  • Revisão é obrigatória: Nunca faça o Merge do seu próprio PR sem que pelo menos uma outra pessoa tenha aprovado.

🚀Extensões Essenciais

Produtividade no Front-End

O VS Code se torna uma ferramenta de alta performance quando você utiliza as extensões certas para o seu fluxo.

  • GitLens: Fundamental para visualizar o histórico de alterações diretamente na linha do código.
  • Prettier: Formata seu HTML, CSS e JavaScript/TypeScript automaticamente ao salvar. Essencial para manter o código limpo.
  • ESLint: Identifica erros de sintaxe e más práticas no seu código em tempo real.
  • Live Share: Permite programar em dupla (pair programming) em tempo real com colegas da equipe.

Visual e UI

  • Auto Close Tag: Fecha automaticamente as tags HTML/JSX. Economiza muito tempo na criação de componentes.
  • Auto Rename Tag: Se você alterar o nome de uma tag de abertura, ele altera a de fechamento automaticamente.
  • Tailwind CSS IntelliSense: Se estiver usando Tailwind, esta extensão é obrigatória. Ela fornece autocompletar e pré-visualização das classes CSS.
  • Error Lens: Mostra as mensagens de erro diretamente na linha do código, facilitando a correção rápida.

⚠️ Dica de Ouro: Workspace

Evite instalar extensões desnecessárias, pois elas consomem muita memória RAM do seu PC.

  • Use o arquivo .vscode/extensions.json para recomendar as extensões oficiais do projeto.
  • Isso garante que todos na equipe tenham a mesma formatação e as mesmas ferramentas, evitando conflitos de estilo no código.

🎯Desafio Final: O Repositório Base

Missão Prática: Ciclo de Vida

  1. Identidade: Configure seu user.name e user.email.
  2. Inicialização: Crie um repositório no GitHub, clone-o na sua máquina e entre na pasta.
  3. Branching: Crie e troque para uma branch chamada feat/estrutura-inicial.
  4. Trabalho: Mova os arquivos da capacitação para a pasta do repositório.
  5. Ciclo: Adicione os arquivos (add), crie o commit padrão (feat: adiciona arquivos da capacitação) e envie para o GitHub (push).
  6. Integração: Abra um Pull Request no GitHub mesclando sua branch na main.
